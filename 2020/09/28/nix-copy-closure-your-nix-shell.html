nix-copy-closure your nix-shell

Published 2020-09-28 on Farid Zakaria's Blog — Permalink

This is a synopsis of https://github.com/NixOS/nix/issues/1985. I recommend reading it for additional detail. Many thanks to contributors on the issue like Infinisil for adding context.

Someone reached out to me over e-mail to discuss my previous post on caching your nix-shell.

‚ÄúWhat I wish to do is to copy a particular development environment (nix-shell) from A to B, so that I could run nix-shell on server B. Server B is only accessible through SSH and does not have Internet access.‚Äù

Seems straightforward, so let‚Äôs investigate. Let‚Äôs use a very basic shell.nix file.

Very important we make sure to pin nixpkgs. In the example below I do it inline in the Nix expression but you can also use niv or pin the channel to a particular commit.

let
  nixpkgs = import (builtins.fetchTarball {
    name = "nixos-unstable-2020-09-24";
    url =
      "https://github.com/nixos/nixpkgs/archive/5aba0fe9766a7201a336249fd6cb76e0d7ba2faf.tar.gz";
    sha256 = "05gawlhizp85agdpw3kpjn41vggdiywbabsbmk76r2dr513188jz";
  }) { };
in with nixpkgs;
with stdenv;
with stdenv.lib;
mkShell {
  name = "example-shell";
  buildInputs = [ hello ];
  shellHook = ''
    export MESSAGE="$(hello)";
  '';
}

Let‚Äôs see it in action:

‚ùØ nix-shell shell.nix

[nix-shell:~/code/nix/playground]$ echo $MESSAGE
Hello, world!

Recently an improvement #95536 by Infinisil made the workflow to discover the transitive runtime closure much simpler than what I had blogged about.

‚ú® A new attribute for derivations inputDerivation is introduced that is always buildable and whose runtime dependences are it‚Äôs build dependencies; exactly what we need for nix-shell!

‚ùØ nix-build --no-out-link shell.nix -A inputDerivation
these derivations will be built:

We now have a store path /nix/store/rw6i1wk9iv0286xi2b6kpw4ynk4pldyh-example-shell which we can check the immediate dependencies.

The below is simply the immediate dependencies and not the full transitive closure for brevity.

I will now copy the shell.nix file to my other machine & copy the transitive closure to it.

‚ùØ scp shell.nix machine-b:~

‚ùØ nix-copy-closure --to machine-b \
    /nix/store/rw6i1wk9iv0286xi2b6kpw4ynk4pldyh-example-shell
copying 36 paths...

Let‚Äôs hop on machine B and create a network namespace to pretend we do not have Internet access.

A network namespace is logically another copy of the network stack, with its own routes, firewall rules, and network devices.

‚ùØ ssh machine-b

# since we won't have any Internet access, hydrate the cache
# with our nixpkgs version
‚ùØ nix-prefetch-url --unpack \
https://github.com/nixos/nixpkgs/archive/5aba0fe9766a7201a336249fd6cb76e0d7ba2faf.tar.gz \
--name "nixos-unstable-2020-09-24"

‚ùØ sudo ip netns add nixshell
# enter the namespace
‚ùØ sudo ip netns exec nixshell su $USER -c zsh

# let's confirm we do not have Internet access
‚ùØ ping google.com
ping: google.com: Temporary failure in name resolution

Let‚Äôs fire up our nix-shell and see if it works.

# don't forget we are within our network namespace
# without access to the Internet
‚ùØ nix-shell shell.nix
bash: cannot set terminal process group (-1): Inappropriate ioctl for device
bash: no job control in this shell

[nix-shell:~]$ echo $MESSAGE
Hello, world!

üéÜ Huzzah! We have now copied over our development environment hermetically with the help of Nix. This is a great demonstration of the power of Nix & reproducibility.

An important take-away though is that it‚Äôs important to make sure we pin the exact version of nixpkgs otherwise the nix-shell may calculate different hashes.
